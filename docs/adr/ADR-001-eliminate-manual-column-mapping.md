# ADR-001: إلغاء واجهات اختيار أعمدة الربط اليدوي (Manual Column Mapping)

هذه الوثيقة سجل قرار معماري (Architecture Decision Record) مستقل — لا تُعدَّل بنودها إلا عبر ADR جديد يُلغيها أو يُعدِّلها صراحةً. لا تُعتبر جزءًا من RIE Constitution.

---

## ١. السياق (Context)

قبل اعتماد FSOS Canonical Database v1.0 وبناء Relationship Intelligence Engine (RIE)، لم تكن المنصة تمتلك مخططًا بيانيًا ثابتًا (Fixed Schema). كل ملف Excel يرفعه المستخدم كان يُعامَل كهيكل حر عشوائي الرؤوس (Variable Headers)، لذلك احتاجت كل شاشة تحليلية إلى واجهة يدوية تسأل المستخدم: "أي عمود يمثل CustomerCode؟ أي عمود يمثل Latitude؟ أي عمود يمثل RouteID؟"... إلخ (Manual Mapping)، مع محاولات مساعدة لاحقة (Guess Columns، Advanced Mapping toggle).

بعد اعتماد Canonical Database v1.0 (١٩ كيانًا بأسماء وحقول ثابتة) وبناء المكونات التشغيلية الخمسة لـ RIE (Graph Builder، Navigation Engine، Query Execution Engine، Business Rules Engine، Integration Layer/RieFacade) — تم إجراء بحث شامل (راجع تقرير "تقييم إلغاء اختيار أعمدة الربط") حدَّد كل الشاشات التي لا تزال تعتمد على هذا النمط اليدوي:

- صفحة الملفات: Hierarchy Columns Editor، Route Hierarchy Config Editor
- New Customer / Geo Intelligence (تحليل + توسّع)
- Customer Comparison
- Heat Map
- Route Planning
- Customer Similarity
- Visit Efficiency
- Team Performance
- Sales Growth (SGI)
- مسار استيراد Targets من ملف (كود غير مستخدم فعليًا من أي شاشة)

خلص التقرير إلى أن معظم هذه الأعمدة (مفاتيح الربط والهوية والموقع والمبلغ الأساسي) أصبحت جزءًا من Canonical Schema ولا يوجد سبب معماري حقيقي لبقائها قابلة للاختيار اليدوي، باستثناء حالتين محدودتين (اختيار بُعد التحليل التحليلي `scopeColumn`/`labelColumn`، والملفات الثانوية غير المصنَّفة) — وهما موضوع خارج نطاق هذا القرار وسيُعاد تقييمهما لاحقًا بشكل منفصل.

---

## ٢. القرار (Decision)

يُعتمد رسميًا كسياسة معمارية ملزمة لكل التطوير القادم على منصة FSOS:

1. **إلغاء كامل ونهائي** لجميع واجهات اختيار أعمدة الربط من المنتج النهائي: لا Manual Mapping، لا Advanced Mapping، لا Guess Columns، بأي شكل من الأشكال.
2. **جميع الشاشات** (بلا استثناء) تعتمد حصريًا على **Canonical Database + RIE** كمصدر وحيد لتحديد الأعمدة/الكيانات/العلاقات — لا يوجد أي شاشة تقرأ ملفًا وتفسّر أعمدته بنفسها.
3. **المصدر الوحيد المقبول للبيانات هو FSOS Import Template.** لا يوجد دعم لملفات أو هياكل غير قياسية أيًا كان مصدرها أو حجمها أو أهميتها للعميل.
4. **آلية الرفض:** أي ملف مرفوع لا يطابق الـ Import Template يُرفض مباشرة، مصحوبًا بـ **Validation Report** يوضح بدقة الحقول أو القيم غير المطابقة وسبب الرفض — **بدون أي بديل لاختيار الأعمدة يدويًا** كمخرج للمستخدم من هذا الرفض. المسار الوحيد لتصحيح الملف هو تعديله ليطابق القالب وإعادة رفعه.
5. **تصنيف Legacy:** كل واجهات اختيار الأعمدة الحالية (المذكورة في القسم ١) تُصنَّف رسميًا Legacy اعتبارًا من تاريخ هذا القرار، وتُحذف تدريجيًا ضمن خطة Migration بعد اكتمال ربط كل شاشة بـ RIE — وليس إلغاءً فوريًا دفعة واحدة، لتفادي كسر الشاشات القائمة قبل جاهزية البديل.

---

## ٣. المصدر المعتمد (Source of Truth) — تأكيد

تم التحقق (SHA-256 checksum + مقارنة كاملة لكل الأوراق) أن ملف `FSOS_Canonical_Database_v1.0.xlsx` المستخدم في بناء RIE منذ البداية (Constitution vNext، Relationship Registry، Query Planner، Graph Builder) **مطابق بايت لبايت** لأي نسخة لاحقة رُفعت بنفس الاسم. لا توجد ولم تُستخدم أي نسخة قديمة بـ ١١ جدولًا في أي مرحلة من هذا العمل. النموذج المعتمد دائمًا هو **١٩ كيانًا كانونيًا**.

---

## ٤. الأثر (Consequences)

**إيجابي:**
- اختصار كبير في خطوات الإعداد لكل شاشة (لا قوائم منسدلة، رفع مباشر ومتابعة).
- إزالة مصدر رئيسي لخطأ المستخدم (اختيار عمود خطأ).
- ثبات تام في تفسير البيانات عبر كل الشاشات لأنها كلها تمر من نفس المصدر (RIE).

**تكلفة/متطلبات تنفيذية:**
- يجب تصميم واعتماد **FSOS Import Template** رسميًا لكل Dataset قبل تنفيذ أي حذف لواجهة قائمة.
- يجب بناء طبقة **Import Validation** (رفض + Validation Report) قبل تعطيل أي مسار Manual Mapping حالي — لا يجوز حذف البديل القديم قبل جاهزية البديل الجديد فعليًا.
- الشاشات العشر المذكورة في القسم ١ يجب ربطها فعليًا بـ RIE (عبر Entity Provider / RieFacade) واحدة تلو الأخرى — هذا الربط غير موجود حاليًا لأي منهم.
- حالة خاصة يجب حلها أولًا: كيان Employees له تمثيلان متوازيان حاليًا (جدول Prisma حقيقي + Dataset Excel مرفوع) — يجب حسم مصدر الحقيقة الواحد قبل ترحيل Hierarchy Columns Editor / Route Hierarchy Config Editor تحديدًا.

**خارج نطاق هذا القرار (تُعالج لاحقًا بقرار منفصل عند الحاجة):**
- `scopeColumn` و`labelColumn` (اختيار تحليلي وليس مفتاح ربط).
- الملفات الثانوية الاختيارية غير المصنَّفة كـ Dataset كانوني.
- مشاكل Data Normalization / جودة القيم داخل الأعمدة (مثل اختلاف تنسيق `RT001` مقابل `RT-001`) — هذه بطبيعتها منفصلة عن Column Mapping حسب القرار السابق المتفق عليه، وتُعالَج ضمن طبقة Data Normalization مستقلة، لا داخل واجهات الشاشات.

---

## ٥. الحالة التنفيذية

**لم يبدأ أي تنفيذ لهذا القرار بعد.** هذه الوثيقة توثيق للقرار المعماري فقط، بانتظار تكليف رسمي منفصل بالبدء في:
1. تصميم FSOS Import Template لكل Dataset.
2. بناء طبقة Import Validation.
3. خطة Migration التفصيلية لربط كل شاشة بـ RIE وحذف واجهاتها القديمة.
