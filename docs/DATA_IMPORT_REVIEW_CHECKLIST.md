# طريقة مراجعة تحويل بيانات الشركات إلى القالب القياسي (FSOS Canonical Database)

هذا المستند هو الطريقة الثابتة اللي نتبعها كل مرة شركة تديني ملف إكسل بشكل
مختلف عن قالبنا، وعايزين نحوّل بياناته لشكل FSOS القياسي (19 شيت، انظر
`FSOS_Canonical_Database_v1.0.xlsx`) قبل ما نرفعه على المنصة.

## ليه المراجعة اليدوية دي مهمة، مش بس الفحص التلقائي بتاع المنصة

المنصة نفسها فيها بوابة فحص تلقائية (`ImportValidationService`) بتشتغل
لحظة رفع أي ملف. لكن حسب قرار معماري متعمد (**ADR-002 — Structure-Only
Import Validation**, 2026-07-19)، الفحص التلقائي ده **بيتأكد بس من شكل
الملف**:

- الأعمدة المطلوبة موجودة (`HEADER-001`).
- مفيش أعمدة غريبة مش معروفة (`HEADER-002`).

**الفحص التلقائي مبيتأكدش من**: نوع البيانات جوه العمود (تاريخ حقيقي ولا
نص؟)، القيم المسموحة (Enum زي Status)، الحدود الرقمية (Min/Max)، الترابط
بين الشيتات (FK)، ولا التكرار في المفاتيح الأساسية. القرار المعماري قال
صراحة إن ده مسؤولية الشركة الرافعة للملف، مش حاجة المنصة بترفضها. يعني:
**لو التحويل بتاعنا فيه غلطة في نوع أو ترابط البيانات، المنصة مش هتوقفها
تلقائي — إحنا لازم نمسكها إحنا قبل الرفع.** هنا بالظبط أهمية المراجعة دي.

## خطوات التحويل (قبل المراجعة)

1. **فهم ملف الشركة** — الشيتات، الأعمدة، وشكل كل عمود (حتى لو كل حاجة
   في شيت واحد بدل 19، أو أسامي الأعمدة مختلفة تمامًا).
2. **بناء خريطة تحويل (Mapping)** — كل عمود عندهم يقابل أنهي عمود عندنا،
   وأي تحويل مطلوب (صيغة تاريخ، فصل/دمج حقول، توليد أكواد ناقصة).
3. **تنفيذ التحويل** فعليًا وكتابة الناتج في شكل القالب القياسي.
4. **تشغيل قائمة المراجعة أدناه** على الناتج، قبل ما يترفع على المنصة.

## قائمة المراجعة الصارمة

### 1. مطابقة الأعداد (Row Count Reconciliation)
عدد صفوف كل شيت في ملف الشركة الأصلي = عدد الصفوف المقابلة في الناتج
(بعد استبعاد أي صفوف استُبعدت عن قصد، ولازم توثيق سبب الاستبعاد صراحةً —
مفيش صف يختفي بصمت).

### 2. سلامة الترابط بين الشيتات (Referential Integrity)
كل قيمة مفتاح أجنبي مذكورة في شيت فرعي، لازم تكون موجودة فعليًا في الشيت
الأساسي المقابل لها:
- `CustomerCode` في Invoices/Visits/Collections/Returns → لازم موجود في Customers.
- `ProductCode` في Invoice Items/Price List/Van Loads/Van Inventory → لازم موجود في Products.
- `RouteID` في أي شيت معاملات → لازم موجود في Routes.
- `EmployeeID` في Route Assignments/DirectManagerID → لازم موجود في Employees.
- `BranchID` → لازم موجود في Branches، و`RegionID` بتاع البرانش → موجود في Regions.
- `InvoiceNo` في Invoice Items/Collections/Returns → لازم موجود في Invoices.
- `PriceListCode`/`DefaultPriceListCode` → لازم موجود في Price List.

**مفيش كود "يتيم" (orphan)** — أي قيمة FK مالهاش أصل في الشيت الأساسي
بتتسجل كخطأ ولازم تتحل قبل الرفع.

### 3. اكتمال الحقول الإلزامية (Required Field Completeness)
كل عمود إلزامي في القالب القياسي (زي `CustomerCode`, `InvoiceNo`,
`ProductCode`, إلخ) لازم يكون مليان في كل صف — مفيش خانة فاضية كان
المفروض تتملي.

### 4. الأنواع والصيغ (Type & Format Checks)
- التواريخ لازم تتفهم كـ Date حقيقي، مش نص شكله تاريخ.
- الأرقام (كمية، سعر، مبلغ) لازم تكون رقم فعلي، مش نص فيه فواصل/رموز عملة.
- الحقول من نوع Enum (زي `Status`, `PaymentMethod`, `InvoiceType`) لازم
  قيمتها من ضمن القيم المسموحة بالظبط (حساسة لحالة الأحرف/التطابق التام).

### 5. الحدود المنطقية (Range Sanity)
أرقام زي الكمية، السعر، نسبة الضريبة، حدود الائتمان — لازم تكون في مدى
منطقي (مفيش كمية سالبة، مفيش نسبة ضريبة أكبر من 100%، إلخ). مش لازم نطاق
دقيق زي القالب الرسمي، بس فحص "هل الرقم ده معقول" كخط دفاع أول.

### 6. التكرار في المفاتيح الأساسية (Uniqueness)
كل Primary Key (زي `CustomerCode`, `ProductCode`, `InvoiceNo`+`LineNo`)
لازم يكون فريد في شيته — مفيش تكرار.

### 7. فحص عيني على عينة (Spot Check)
مراجعة يدوية لعدد صغير من الصفوف (10-20 صف عشوائي من كل شيت رئيسي) —
مقارنة مباشرة بين المصدر الأصلي والناتج، للتأكد إن التحويل مش بس "مر
الفحوصات الآلية" لكن كمان منطقي وصحيح فعليًا.

## تقرير المراجعة (Output لكل عملية تحويل)

بعد كل تحويل، بتتبني ورقة ملخص (Excel أو نص) تتضمن:

| البند | القيمة |
|---|---|
| عدد الشيتات المحوّلة | |
| إجمالي الصفوف (دخول) | |
| إجمالي الصفوف (خروج) | |
| صفوف مُستبعدة + السبب | |
| أخطاء ترابط (FK) مكتشفة | |
| أخطاء نوع/صيغة مكتشفة | |
| تكرارات مكتشفة | |
| تحذيرات (مش أخطاء، بس تستاهل مراجعة) | |
| القرار النهائي | جاهز للرفع / محتاج تصحيح |

## معيار "جاهز للرفع"

الملف يُعتبر جاهز للرفع على المنصة لما:
- صفر أخطاء ترابط (FK) يتيمة.
- صفر حقول إلزامية فاضية.
- صفر تكرارات في المفاتيح الأساسية.
- كل القيم من نوع Enum مطابقة تمامًا للقيم المسموحة.
- تقرير المراجعة مكتوب ومراجَع من أحمد شخصيًا قبل الرفع النهائي.
